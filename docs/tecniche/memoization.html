Palestra di Algoritmi

Memoization

La memoization è una tecnica che permette di migliorare soluzioni progettate tramite backtracking (quando possibile).

Esempio - Domino (non massimale)

Il domino è formato da tessere 2x1 (senza numeri).
Si scriva un algoritmo che, letto in input un numero N, dica quanti modi possibili ci sono per poter posizionare delle tessere in modo da coprire interamente un rettangolo 2xN.

Sia dato in input il numero 4, le possibilità sono:

┌─┬─┬─┬─┐ ┌─┬─┬───┐ ┌─┬───┬─┐ ┌───┬─┬─┐ ┌───┬───┐
│ │ │ │ │ │ │ ├───┤ │ ├───┤ │ ├───┤ │ │ ├───┼───┤
└─┴─┴─┴─┘ └─┴─┴───┘ └─┴───┴─┘ └───┴─┴─┘ └───┴───┘

Pertanto la risposta è 5.

È abbastanza semplice scrivere un algoritmo con la tecnica del backtracking.

int domino(int N) {
    // Il rettangolo 2x0 ha come possibilità quella di non mettere alcuna tessera.
    // Il rettangolo 2x1 ha come possibilità quella di mettere una tessera verticale.
    // Precalcoliamo anche per N=1 in modo da non avere problemi quando vogliamo mettere una tessera orizzontale.
    if (N <= 1) return 1;

    // Possiamo mettere una tessera verticale,
    // chiedendoci poi in quanti modi sia possibile riempire il rettangolo rimanente 2x(N-1).
    int numero_modi = domino(N - 1);

    // Oppure possiamo mettere due tessere orizzontali (N >= 2) riducendo la dimensione di 2.
    numero_modi += domino(N - 2);

    // PS: è fibonacci!

    return numero_modi;
}

La complessità di questo algoritmo è data da \(\Theta\left(\varphi^N\right)\), ossia è esponenziale.

La versione lineare iterativa di Fibonacci è abbastanza conosciuta, tuttavia cerchiamo di arrivarci per step.

Problema algoritmo ricorsivo

Simulando la chiamata a domino(4) è possibile osservare come vengano prodotte le seguenti chiamate ricorsive:

  • domino(4) -> 5

    • domino(3) -> 3

      • domino(2) -> 2

        • domino(1) -> 1
        • domino(0) -> 1

      • domino(1) -> 1

    • domino(2) -> 2

      • domino(1) -> 1
      • domino(0) -> 1

Come si può osservare, il problema della soluzione ricorsiva è che vengono risolti degli stessi problemi più volte (domino(2), domino(1) e domino(0)).

Soluzione: memoization

Un modo di risolvere questo problema consiste nel memorizzare i risultati e "riciclarli":

// Ci creiamo una cache di dimensione opportuna (dato dalle constraints nel pdf, oppure la N letta da input).
// Inizializziamo a 0, se il valore è 0 allora sicuramente non lo abbiamo calcolato (Fibonacci ha valori > 0).
vector<int> cache(N + 1, 0);

int domino(int N) {
    // Se lo abbiamo già calcolato allora restituiamo il valore trovato.
    if (cache[N] > 0) return cache[N];

    // Altrimenti proseguiamo con l'algoritmo precedente.
    
    if (N <= 1) return cache[N] = 1;

    // Memorizziamo il valore in modo che sia disponibile per chiamate successive.
    return cache[N] = domino(N - 1) + domino(N - 2);
}

Questo algoritmo si limita a compilare la cache, pertanto la sua complessità è data da \(\Theta\left(N\right)\).
Ogni chiamata alla funzione con parametri differenti corrisponde ad un problema differente (con risposta differente).

In realtà non è propriamente così, non sempre i parametri sono indipendenti.

Per la soluzione iterativa proseguire su Programmazione Dinamica

Esempio - Sommelier

Consideriamo il problema Sommelier proposto al day2.

La soluzione backtracking è la seguente:

vector<int> vini;

/**
 * Funzione che prova a prendere e a non prendere un vino.
 * @param i Indice del vino in arrivo.
 * @param minima_gradazione La minima gradazione che deve avere il vino per poter essere preso (gradazione dell'ultimo vino preso).
 * @return Il numero massimo di vini da `i` in poi che possono essere presi.
 */
int sommelier(int i = 0, int minima_gradazione = 0) {
  // Sono finiti i vini, non ne possiamo prendere altri: "al massimo ne possiamo prendere 0".
  if (i >= vini.size()) return 0;

  int gradazione = vini.at(i);

  if (gradazione < minima_gradazione) {
    // Se la gradazione del vino è troppo bassa allora non ci poniamo nemmeno il problema: non possiamo prenderlo.
    return sommelier(i + 1, minima_gradazione);
  } else {
    // Altrimenti proviamo a vedere quale scelta ci permette di ottenere un risultato migliore.
    return max(
      // Non prendo il vino: passo al successivo e la minima gradazione rimane sempre quella.
      sommelier(i + 1, minima_gradazione),
      // Prendo il vino: salto il vino successivo e i prossimi vini dovranno avere gradazione >= di questo vino. +1 perchè ho preso questo `vino`.
      sommelier(i + 2, vino) + 1
    );
  }
}

In questo problema la funzione prende due argomenti, pertanto la cache dovrà essere bidimensionale (matrice).
Ogni dimensione permette di far spaziare i valori di uno dei parametri.

Anche qui applicare la memoization è abbastanza semplice:

vector<int> vini;

// Il primo indice è relativo a `i`.
// Il secondo indice è relativo a `minima_gradazione`.
// `i` può spaziare da 0 a N con N escluso (N valori).
// `minima_gradazione` può spaziare da 0 a MAX_GRADAZIONE inclusi (MAX_GRADAZIONE+1 valori).
// Inizializziamo a -1 perchè 0 è un valore valido di risposta.
vector<vector<int>> cache(N, vector<int>(MAX_GRADAZIONE + 1, -1));

int sommelier(int i = 0, int minima_gradazione = 0) {
  if (i >= vini.size()) return 0;

  int& result = cache[i][minima_gradazione];
  // Se il valore è già stato calcolato allora restituiamo il valore.
  if (result >= 0) return result;

  // Altrimenti proseguiamo con l'algoritmo.

  int gradazione = vini.at(i);

  if (gradazione < minima_gradazione) {
    // Salviamo il risultato.
    return result = sommelier(i + 1, minima_gradazione);
  } else {
    // Salviamo il risultato.
    return result = max(
      sommelier(i + 1, minima_gradazione),
      sommelier(i + 2, vino) + 1
    );
  }
}

Al peggio verrà compilata tutta la tabella di dimensione N·MAX_GRADAZIONE, la complessità è proprio data dal prodotto fra questi due parametri.
Essendo MAX_GRADAZIONE una costante, la complessità è lineare.

Criticità con la memoria

Fin'ora non abbiamo dato troppa importanza alla questione dei limiti di memoria.
La memoization, tuttavia, può richiedere quantità di memoria importanti, soprattutto con molte dimensioni.

Vedremo con la programmazione dinamica che in alcuni casi è possibile ridurre la dimensione della memoria utilizzata.

In altri casi è opportuno usare le unordered_map quando si può supporre che vengano compilate solo poche celle della matrice.
